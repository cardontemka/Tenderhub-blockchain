# TenderHub — Тендерийн блокчейн систем

Тендерийн (худалдан авах ажиллагааны) үйл явцыг блокчейн дээр **ил тод, шударга**
явуулах систем. Гэрээний нэхэмжлэлийг (санхүүжилтийг) токен болгож, ажил дуусахад
банкны **Escrow** данснаас мөнгө болгон солих зарчмаар ажиллана.

- **Smart contract:** Solidity + Hardhat (`contracts/TenderHub.sol`)
- **Token стандарт:** ERC-1155 (санхүүжилтийн токен + баримтын NFT хоёуланг)
- **Frontend:** Next.js туршилтын орчин (`frontend/`) — Role сольж бүх workflow-г харах

---

## 1. Архитектур

Тендер бүрт **нэг `TenderHub` гэрээ** Захиалагчаар deploy хийгдэнэ. Уг гэрээ нь ERC-1155
бөгөөд хоёр төрлийн токен агуулна:

| Төрөл | ID муж | Тайлбар |
|------|--------|---------|
| **Санхүүжилт / Хөлсний токен** | `1, 2, 3 …` | Багц бүр (урьдчилгаа / шат бүр / арбитрын хөлс) тусдаа ID-тай. **1 токен = 1 ₮**. Escrow данснаас мөнгө авах эрхийг илэрхийлнэ. |
| **Баримт бичгийн NFT** | `≥ 1,000,000,000` | Гэрээ, тайлан, гомдол, тогтоол, зээлийн нөхцөл, өөрчлөлт бүрийг `{hash, uri, timestamp, submitter}`-тэйгээр 1ш токен болгон бэхжүүлнэ. |

### Токений төлвүүд (даалгаврын дагуу)

- **Идэвхтэй / Идэвхгүй** → бүх гэрээний `frozen` тугтай. Арбитр дуудагдсан үед
  `true` болж: шилжүүлэг, солилт боломжгүй, эцсийн хугацааны тоолол зогсоно.
- **Эрхтэй / Эрхгүй** → багц (tranche) бүрийн `entitled`. Шат батлагдахад `true`
  болж мөнгө болгох эрхтэй болно (урьдчилгаа идэвхжих үед шууд эрхтэй).
- **Төлөгдсөн** → солих үед токен **шатаагдана** (burn).

### Шилжүүлгийн гол хязгаарлалт
`_update`-г override хийж: **зөвхөн Захиалагч эсвэл Гүйцэтгэгч** санхүүжилтийн токен
шилжүүлж чадна; гэрээ царцсан үед бүх шилжүүлэг хаагдана; шууд burn хийхийг хориглож
зөвхөн `redeem`-ээр явуулна. Зээлийн барьцааны шилжүүлэг нь гэрээ өөрөө (multisig
батлагдсаны дараа) гүйцэтгэдэг тул дотоод горимоор зөвшөөрөгдөнө.

### Oracle (Банк)
Бодит Escrow данс гэрээнээс гадуур. Тиймээс томилогдсон **oracle** (банк) дараахыг
гэрээнд мэдээлнэ: барьцаа орж ирсэн, Escrow-ийн үлдэгдэл, солих хүсэлтийг гүйцэтгэх.

## 2. 25 функцийн зураглал (даалгавар → код)

| # | Workflow | Гэрээний функц | Хэн |
|---|----------|----------------|-----|
| 1 | Гэрээг deploy хийх | `constructor` | Захиалагч |
| 2 | Санхүүжилт/хугацаа зарлах | `declareFunding` (+ нэмэлт `amendFunding`) | Захиалагч |
| 3 | Гүйцэтгэгч урих | `inviteContractor` | Захиалагч |
| 4 | Гэрээг шалгаж идэвхжүүлэх | `confirmCollateral` (oracle) → `acceptContract` | Oracle → Гүйцэтгэгч |
| 5 | Арбитр урих | `inviteArbiter` | Захиалагч |
| 6 | Арбитрыг батлах | `approveArbiter` | Гүйцэтгэгч |
| 7 | Арбитр үүргээ авах | `arbiterAccept` | Арбитр |
| 8 | Токен барьцаалж зээл хүсэх | `requestLoan` | Гүйцэтгэгч |
| 9 | Зээлийн нөхцөл явуулах | `offerLoan` | Зээл олгогч |
| 10 | Зээлийг хүлээн авах | `acceptLoan` | Гүйцэтгэгч |
| 11 | Токен шилжүүлэхийг зөвшөөрөх (multisig) | `approveLoanTransfer` | Захиалагч |
| 12 | Зээл олгох | `fundLoan` | Зээл олгогч |
| 13 | Зээлээ төлж барьцаа эргүүлэх | `repayLoan` | Гүйцэтгэгч |
| 14 | Барьцааг буцаах | `returnCollateral` | Зээл олгогч |
| 15 | Шат/ажил хүлээлгэн өгөх | `submitDelivery` | Гүйцэтгэгч |
| 16 | Ажил батлаж эрх нээх | `approveDelivery` | Захиалагч |
| 17 | Арбитр дуудах (царцаах) | `callArbiter` | Тал |
| 18 | Арбитрын шийдвэр | `issueRuling` (+ `finalizeRuling` 7 хоног) | Арбитр |
| 19 | Тогтоол хүлээн зөвшөөрөх/давж заалдах | `respondToRuling` | Тал |
| 20 | Давж заалдах | `appeal` | Тал |
| 21 | Шинэ арбитр томилох | `approveArbiter`/`arbiterAccept` + `bindArbiterToDispute` + `fundAppealArbiter` | — |
| 22 | Гэрээнд өөрчлөлт санал болгох | `proposeAmendment` | Тал |
| 23 | Өөрчлөлтийг зөвшөөрөх | `respondToAmendment` | Тал |
| 24 | Арбитр дуудах (өөрчлөлтийн контекст) | `callArbiter` | Тал |
| 25 | Токен тушааж мөнгө болгох | `setBankAccount` + `redeem` + `settleRedemption` (oracle) | Эзэмшигч / Oracle |

## 3. Эхлүүлэх

```bash
# үндсэн фолдер
npm install
npx hardhat compile
npx hardhat test          # 32 тест (цаг урагшлуулах кейсүүд орсон)
```

### Frontend ажиллуулах (visual туршилт)

```bash
# 1) терминал A — локал блокчейн
npx hardhat node

# 2) терминал B — ABI экспортлоод frontend асаах
node scripts/export-abi.js     # (compile хийсний дараа автоматаар бэлэн байдаг)
cd frontend
npm install
npm run dev                    # http://localhost:3000
```

Frontend нь Hardhat node руу шууд (`http://127.0.0.1:8545`) холбогдож, Hardhat-ийн
тогтмол dev акаунтуудыг ашиглана. Зүүн талын самбараас **хэн болж нэвтрэхээ** сонгож
(Захиалагч, Гүйцэтгэгч, Арбитр, Банк/Oracle, Зээл олгогч гэх мэт) бүх workflow-г
дамжуулан туршина.

**Эхлэх дараалал:** `#0 (Захиалагч)` сонгож "Шинэ тендер deploy хийх" → дараа нь
талуудаар нэвтэрч 2→4-р алхмуудыг хийнэ.

## 4. Шийдвэр ба таамаглалууд (Design notes)

- **NFT/IPFS симуляци:** Тайлан/гомдлыг бодит IPFS руу upload хийхгүй; JSON metadata-г
  `data:application/json;base64,…` data-URI болгож, keccak256 hash-тэй нь хамт on-chain
  бэхжүүлнэ (`frontend/lib/nft.js`).
- **Multisig:** Зээлийн барьцаа шилжүүлэхэд Гүйцэтгэгч (`acceptLoan`) + Захиалагч
  (`approveLoanTransfer`) хоёулангийн зөвшөөрөл шаардана.
- **Арбитрын хөлс** нь нийт санхүүжилтээс **тусдаа** (`TrancheKind.ArbiterFee`,
  `totalFinancing`-д нэмэгддэггүй).
- **Маргааны шийдэл (`_applyRuling`)** — талуудын эдийн засгийн үр дагаврын тайлбар нь
  кодын тайлбарт бий. Ялагч = зөв тал; ялагдагч зардлыг үүрнэ. Захиалагч ялбал
  Гүйцэтгэгчийн эрхгүй санхүүжилт Захиалагч руу шилжиж эрх нь нээгдэнэ; Гүйцэтгэгч ялбал
  маргаантай шат түүнд эрхтэй болж, дуудсан тал хөлсөө буцаан токеноор авна. Давж
  заалдсан тохиолдолд бүх арбитрын хөлс хэлхээ эцэслэн зөвшөөрөгдөх хүртэл эрхгүй хэвээр.
- **Контрактын хэмжээ:** Гэрээ 24KB-аас том тул локал сүлжээнд `allowUnlimitedContractSize`
  идэвхжүүлсэн ба frontend дээр deploy хийхдээ тодорхой `gasLimit` өгдөг.
